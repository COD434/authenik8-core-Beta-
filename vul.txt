Findings

  1. High: OAuth token issuance trusts a caller-supplied profile
     src/createAuthenik8.ts:81, src/types/public.ts:47
     issueTokensFromProfile(profile) is public and issues tokens if email_verified is true. If an app route passes user input here, an attacker can mint tokens
     for arbitrary emails/providers.
     Fix: remove this from the public API, or accept only a branded VerifiedOAuthProfile returned by provider handleCallback. Best: provider callback should
     resolve identity and issue tokens internally.

  2. High: Admin middleware is not session-bound
     src/middleware/adminService.ts:33
     requireAdmin accepts any valid JWT with role: "admin". Even when Redis is passed, it only uses Redis for admin actions, not to validate the admin session.
     Revoked/stolen admin JWTs remain valid until expiry.
     Fix: require userId and sessionId, then verify the exact token/session exists in sessions:<userId> before allowing admin access.

  3. High: Plaintext password storage helper
     src/storage/userStorage.ts:6, src/types/storage.ts:4
     register(email, password) passes the raw password into userStore.create.
     Fix: hash with Argon2id or bcrypt before storage, store passwordHash, and remove raw password from persisted User.

  4. High: Refresh flow corrupts the session store shape
     src/auth/jwtAuth.ts:72, src/auth/refreshService.ts:74
     JWTService stores sessions as a Redis hash, but RefreshService writes sessions:<userId> as a plain string. That breaks hgetall, revocation, session
     listing, and session validation after refresh.
     Fix: centralize a SessionStore; refresh should update the same hash format and preserve/rotate sessionId.

  5. Medium: Refresh rotation can overwrite state before detecting replay/race
     src/auth/refreshService.ts:138
     getset writes the new refresh token before checking the previous value. If the previous value is unexpected, the method throws after mutating Redis.
     Fix: replace with an atomic compare-and-set Lua script: only set the new token when current token equals the presented token. On reuse, revoke the token
     family/session.

  6. Medium: Cookie auth without CSRF controls
     src/auth/jwtAuth.ts:121, src/middleware/adminService.ts:16
     Middleware accepts req.cookies.token. If users put auth tokens in cookies, state-changing routes become CSRF-sensitive.
     Fix: disable cookie auth by default, or require CSRF tokens plus SameSite=Lax/Strict, HttpOnly, Secure.

  7. Medium: Custom token bucket is non-atomic
     src/security/limiter.ts:21
     Rate limiting does read-then-write with separate Redis operations, so concurrent requests can overrun limits.
     Fix: use a Lua script or rate-limiter-flexible for OTP/login limiters too.

  8. Medium: Redis URL/TLS config is validated but ignored
     src/redis/redisService.ts:48, src/redis/redisService.ts:75
     rediss:// is accepted, but setupRedis constructs Redis from host/port/password and ignores url, so TLS expectations can silently fail.
     Fix: if config.url exists, pass it to new Redis(config.url, options) and preserve TLS settings.

  Good signs

  - npm audit --omit=dev and full npm audit both returned 0 vulnerabilities.
  - Google OAuth verifies ID token audience, issuer, email, and email verification.
  - OAuth state uses strong randomness.

  I’d fix items 1-4 before publishing another SDK version. Those affect authentication correctness directly.
